## Контрольный вопрос №3
### Реализация AUTO_INCREMENT в MySQL

Для автоматической генерации первичного ключа добавьте атрибут AUTO_INCREMENT
к целочисленному столбцу, объявленному как PRIMARY KEY:

```text
CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL
);
```

При вставке без явного указания id MySQL автоматически подставит
следующее значение внутреннего счётчика.

### Почему AUTO_INCREMENT не подходит для распределённых систем

- 🔒 **Централизованный счётчик** — создаёт bottleneck и единую точку отказа.
- 🔄 **Проблемы репликации** — задержки между узлами могут приводить к конфликтам, дубликатам ID или разрывам нумерации.
- 🌐 **Отсутствие глобальной уникальности** — независимые инстансы БД не синхронизируют счётчики без дополнительной координации.
- ⚡ **Блокировки метаданных** — при высокой конкуренции InnoDB сериализует вставки, снижая пропускную способность кластера.

### Альтернативы для распределённых архитектур

- **UUID v4 / v7** — гарантированная уникальность; v7 дополнительно сортируется по времени.
- **Snowflake ID** — 64-битные, временные метки + ID узла, генерируются без БД, идеально для шардинга.
- **Настроенные последовательности** — каждый узел использует свой шаг и смещение (OFFSET/INCREMENT), например: узел 1 → 1, 4, 7..., узел 2 → 2, 5, 8....
- **Внешние генераторы ID** — Redis, ZooKeeper, или специализированные сервисы (Meituan Leaf, TikTok Snowflake).
