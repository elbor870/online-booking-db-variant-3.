## 🔐 Подготовленные выражения (Prepared Statements) и защита от SQL-инъекций
### 📚 Что такое подготовленные выражения?
**Подготовленное выражение** — это механизм работы с базой данных, при
котором **SQL-запрос и данные передаются отдельно.**

**Процесс выполнения в 2 этапа:**

```text
Этап 1: ПОДГОТОВКА (Prepare)
┌─────────────────────────────────┐
│ Вы отправляете в БД "шаблон"     │
│ запроса с плейсхолдерами:        │
│                                 │
│ "SELECT * FROM clients          │
│  WHERE phone = ?"               │
│                                 │
│ БД парсит, компилирует и         │
│ сохраняет план выполнения.       │
└─────────────────────────────────┘
                ↓
Этап 2: ВЫПОЛНЕНИЕ (Execute)
┌─────────────────────────────────┐
│ Вы отправляете только данные:   │
│                                 │
│ ['+79991234567']                │
│                                 │
│ БД подставляет данные в         │
│ уже скомпилированный план.      │
│ Данные НИКОГДА не становятся    │
│ частью SQL-кода.                │
└─────────────────────────────────┘
```
## ⚠️ Как работает SQL-инъекция (без подготовленных выражений)
### ❌ Уязвимый код (конкатенация строк):

```php
// ЗЛОУМЫШЛЕННИК вводит в поле телефона:
$phone = "+79991234567' OR '1'='1";

// Разработчик пишет "наивный" запрос:
$sql = "SELECT * FROM clients WHERE phone = '" . $phone . "'";

// В итоге выполняется ЗАПРОС:
// SELECT * FROM clients WHERE phone = '+79991234567' OR '1'='1'
//                                ↑                    ↑
//                        строка закрыта     новое условие: ВСЕГДА ИСТИНА

// Результат: злоумышленник получает ВСЕХ клиентов из базы! 🚨
```

## ✅ Как подготовленные выражения блокируют инъекции
### ✔️ Безопасный код с PDO (как в моём репозитории):

```php
// Файл: src/Repository/ClientRepository.php

public function findByPhone(string $phone): ?array
{
    // 1. Шаблон запроса с плейсхолдером (?)
    $stmt = $this->pdo->prepare("SELECT * FROM {$this->table} WHERE phone = ?");
    
    // 2. Данные передаются ОТДЕЛЬНО в execute()
    $stmt->execute([$phone]);
    
    return $stmt->fetch() ?: null;
}
```

### Что происходит внутри при $phone = "+79991234567' OR '1'='1":

```text
Этап 1: prepare()
┌─────────────────────────────────┐
│ БД получает и компилирует:      │
│ "SELECT * FROM clients          │
│  WHERE phone = ?"               │
│                                 │
│ Плейсхолдер ? — это "ячейка"    │
│ для данных, а не часть кода.    │
└─────────────────────────────────┘

Этап 2: execute(['+79991234567\' OR \'1\'=\'1'])
┌─────────────────────────────────┐
│ БД подставляет значение как     │
│ БУКВАЛЬНУЮ СТРОКУ, экранируя    │
│ спецсимволы:                    │
│                                 │
│ phone = '+79991234567\' OR \'1\'=\'1' │
│                                 │
│ Это поиск телефона, который     │
│ буквально равен этой строке.    │
│ Результат: ПУСТО (нет такого клиента) │
└─────────────────────────────────┘
```
> 🔑 Ключевой момент: Данные никогда не интерпретируются как SQL-код.
Даже если злоумышленник введёт '; DROP TABLE clients;--, это будет
просто искаться как телефон с таким "странным" номером.

## 📦 Примеры из моего кода (DAL для автосервиса)
### 1. Абстрактный репозиторий: динамическая фильтрация

```php
// Файл: src/Repository/AbstractRepository.php

public function findAll(array $where = [], array $params = [], ...): array
{
    $sql = "SELECT * FROM {$this->table}";
    $conditions = [];
    $bindings = $params;

    // Формируем условия с плейсхолдерами
    foreach ($where as $col => $val) {
        $conditions[] = "$col = ?";  // ← ? это плейсхолдер
        $bindings[] = $val;           // ← данные отдельно
    }
    
    if ($conditions) {
        $sql .= " WHERE " . implode(" AND ", $conditions);
    }

    // Подготовка и выполнение РАЗДЕЛЕНЫ
    $stmt = $this->pdo->prepare($sql);  // ← компиляция шаблона
    $stmt->execute($bindings);           // ← подстановка данных
    return $stmt->fetchAll();
}
```
### Пример использования:

```php
// Злоумышленник пытается внедрить код через last_name:
$clients = $clientRepo->findAll(
    where: ['last_name' => "Иванов'; DROP TABLE clients;--"]
);

// Безопасный итоговый запрос:
// SELECT * FROM clients WHERE last_name = ?
// Данные: ["Иванов'; DROP TABLE clients;--"]
// Результат: поиск фамилии, которая буквально равна этой строке → пусто
```
### 2. Вставка с транзакцией (сложный случай)

```php
// Файл: src/Repository/AppointmentRepository.php

public function createAppointment(...): int
{
    $this->pdo->beginTransaction();
    try {
        // Подготовка запроса
        $stmt = $this->pdo->prepare("
            INSERT INTO appointments 
            (client_id, car_id, service_id, appointment_datetime, status)
            VALUES (?, ?, ?, ?, ?)
        ");
        
        // Выполнение с данными
        $stmt->execute([$clientId, $carId, $serviceId, $datetime, $status]);
        
        $this->pdo->commit();
        return (int)$this->pdo->lastInsertId();
    } catch (\PDOException $e) {
        $this->pdo->rollBack();
        throw new RepositoryException("Ошибка: " . $e->getMessage());
    }
}
```
## 🛡️ Дополнительные меры защиты в моём коде
### 1. Белый список для сортировки (защита от инъекций в ORDER BY)

```php
// Файл: src/Repository/AbstractRepository.php

protected array $allowedSortColumns = ['client_id', 'last_name', 'created_at'];

// В методе findAll():
if (!empty($orderBy)) {
    $validClauses = [];
    foreach ($orderBy as $col => $dir) {
        // Проверяем, что имя столбца в белом списке
        if (in_array($col, $this->allowedSortColumns, true)) {
            $direction = strtoupper($dir) === 'DESC' ? 'DESC' : 'ASC';
            $validClauses[] = "$col $direction";  // ← безопасно, $col из белого списка
        }
    }
    // ...
}
```
> ⚠️ Важно: Имена столбцов нельзя передавать через плейсхолдеры.
> Поэтому мы валидируем их через in_array() по белому списку.

### 2. Настройки PDO для максимальной безопасности

```php
// Файл: src/Database.php

$options = [
    // Выбрасывать исключения при ошибках (не скрывать проблемы)
    PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
    
    // Возвращать результаты как ассоциативные массивы
    PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,
    
    // ❗ ОТКЛЮЧИТЬ эмуляцию подготовленных выражений
    // Это заставляет MySQL использовать настоящие подготовленные запросы
    PDO::ATTR_EMULATE_PREPARES => false,
];
```
### 🧪 Демонстрация: уязвимый код vs. безопасный

```php
<?php
// === ❌ УЯЗВИМЫЙ КОД (никогда не делайте так!) ===
$phone = $_GET['phone']; // пользователь вводит: ' OR '1'='1
$sql = "SELECT * FROM clients WHERE phone = '$phone'";
$result = $pdo->query($sql); // 💥 SQL-инъекция возможна!

// === ✅ БЕЗОПАСНЫЙ КОД (как в моём репозитории) ===
$phone = $_GET['phone']; // даже если введено: ' OR '1'='1
$stmt = $pdo->prepare("SELECT * FROM clients WHERE phone = ?");
$stmt->execute([$phone]); // 🔐 Данные экранированы, инъекция невозможна
$result = $stmt->fetchAll();
```
### 🔑 Ключевые выводы
1. Подготовленные выражения — это не просто «экранирование кавычек».
Это разделение кода и данных на уровне протокола связи с БД.

2. Плейсхолдеры или :name — это не строки, которые потом заменяются.
Это «слоты», в которые БД подставляет данные уже после компиляции запроса.

3. PDO::ATTR_EMULATE_PREPARES => false — критически важная настройка. При
true PDO эмулирует подготовку в PHP, что менее безопасно.

4. **Имена таблиц и столбцов** нельзя передавать через плейсхолдеры. Для
них используйте **белый список** (in_array()), как я сделал в $allowedSortColumns.

5. **Все пользовательские данные** (из $_GET, $_POST, API, заголовков) должны
проходить через подготовленные выражения — без исключений.

> 💡 Правило одного предложения:
«Если данные пришли от пользователя — они должны попасть в запрос
> только через prepare() + execute(), никогда через конкатенацию строк.»

Этот подход реализован во всех методах моих репозиториев: findByPhone(),
createAppointment(), findAll() и других. 🔐🚗
